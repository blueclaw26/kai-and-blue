環境分離の設計と実装

コメントアウトから卒業し、production/staging分離に至るまで

ここで学ぶこと: 環境分離 環境変数 PaaS CI/CD DNS

きっかけ — コメントアウトの限界

最初のやり方は単純だった。テスト時にはコードの一部をコメントアウトし、本番デプロイ時に戻す。それだけ。

// テスト時はここをコメントアウト
// if (!verifyIdentity(user)) return res.status(403).send('認証失敗');

// テスト時はこっちを有効に
res.status(200).send('OK'); // 常にパス

問題は明らかだった。

  • 本番デプロイ時にコメントを戻し忘れる → テスト設定のまま公開
  • どこをコメントアウトしたか覚えていない
  • 複数ファイルにまたがるとさらに危険

決定打はAIサブエージェントによるリファクタリング中の事故だった。サブエージェントは本番DBと開発DBの区別がつかない。大規模なコード変更の過程で、本番データベースが初期化されてしまった。

構造的に事故を防ぐ仕組みが必要だと痛感した。

環境変数という解決策

環境変数を使えば、コードを一切変えずに振る舞いを変えられる。

// コード内での分岐
if (process.env.SKIP_EKYC === '1') {
  // 本人確認をスキップ（staging環境のみ）
} else {
  // 通常の本人確認フロー（production環境）
}

テスト用の環境変数はstaging環境にだけ設定する。productionには設定しない。つまり、productionでは自動的に安全な方に倒れる。

テスト用環境変数

変数名用途productionstaging
SKIP_EKYC本人確認スキップ未設定（通常フロー）1
SKIP_COOLDOWNクールダウン無効未設定（65日制限）1
RESET_TOKENデータリセットAPI未設定（API無効）設定済み

インフラの変遷

Phase 1

VPS + SQLite

手動デプロイ。環境は1つ。sshでサーバーに入ってgit pullして再起動。

開発PC → ssh → VPS (SQLite)
Phase 2

PaaS + PostgreSQL

GitHubへのpushで自動デプロイ。DBもマネージドに。でもまだ環境は1つ。

GitHub → auto → PaaS (PostgreSQL)
Phase 3

PaaS Pro Plan + Environments

production/staging分離。ブランチごとに自動デプロイ先が分かれる。

GitHub main → auto → Production
GitHub dev → auto → Staging

ブランチ戦略

gitのブランチとデプロイ先を対応させる。

  • mainブランチ → production環境に自動デプロイ
  • devブランチ → staging環境に自動デプロイ

開発はdevで行い、確認できたらmainにマージ。AI（サブエージェント）もdevブランチで作業し、staging環境でテスト、問題なければmainにマージする。

これにより、AIが本番を直接触ることがなくなる。構造的に安全。

Kaiが学んだこと

  • 「環境分離は大企業だけのもの」は間違い。個人開発でも必須
  • 事故が起きてから対策するのではなく、構造的に事故を防ぐ
  • 環境変数は「設定のコード外出し」という単純な概念
  • PaaSのEnvironments機能で、個人でも簡単に実現できる
  • AIと開発する場合、AIが本番を壊せない構造が特に重要
